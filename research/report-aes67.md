# ST Lab AES67 Truth Report

## 1. Executive Summary

AES67-2023 is the current AES standards-store entry for “AES Standard for Audio Applications of Networks - High-performance streaming audio-over-IP interoperability,” with a printing date shown by AES as 2023-06-01 on the store page and 2023-12-18 in the official preview PDF metadata; this report treats that date difference as a source conflict rather than silently reconciling it ([AES Standards Store](https://aes.org/publications/standards-store/), [AES67-2023 preview](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)). The readable AES67-2023 preview states that the standard defines an interoperability mode for synchronization, encoding, transport, and connection management of high-performance audio over IP networks, and it states that high-performance audio implies linear PCM coding at 44.1 kHz and higher with 16-bit and higher resolution and considers latency performance of 10 ms or less ([AES67-2023 preview, Clause 1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).

The public AES67-2023 preview exposes the scope, normative references, table of contents, foreword, and selected change notes, but it does not expose the full normative text of Clauses 3-10 or Annex G PICS, so AES67-specific “shall” requirements that depend on hidden clause text are marked **Unverified** unless they are visible in the preview or corroborated only as secondary guidance ([AES67-2023 preview, preview omissions](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)). The strongest publicly clause-backed requirements for ST Lab modeling come from AES67’s visible normative references, including RTP in RFC 3550, RTP/AVP in RFC 3551, RTP payload format for L24 in RFC 3190, SDP in RFC 8866, RTP clock-source signaling in RFC 7273, and PTP in IEEE 1588-2008 as listed by AES67 ([AES67-2023 preview, Clause 2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 3550](https://www.rfc-editor.org/rfc/rfc3550.html), [RFC 3551](https://www.rfc-editor.org/rfc/rfc3551.html), [RFC 3190](https://www.rfc-editor.org/rfc/rfc3190.html), [RFC 8866](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 7273](https://www.rfc-editor.org/rfc/rfc7273.html), [IEEE 1588-2008](https://standards.ieee.org/standard/1588-2008.html)).

For ST Lab, AES67 should be modeled as a transport-interoperability profile rather than a full device-control ecosystem, because the AES67 preview scope is limited to synchronization, encoding, transport, and connection management, while vendor/secondary guidance states that stream/device discovery and the method of conveying SDP are deliberately not mandated by AES67 ([AES67-2023 preview, Clause 1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)). ST Lab should therefore require explicit project assumptions for discovery method, controller behavior, multicast management, PTP profile/version handling, SDP exchange method, receiver buffering margin, and DSCP/QoS policy whenever those values are not established by a visible normative citation ([AES67-2023 preview, Clauses 8-10 and Annexes B/E/G table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 7273, Sections 4-7](https://www.rfc-editor.org/rfc/rfc7273.html), [RFC 8866, Sections 5-6](https://datatracker.ietf.org/doc/html/rfc8866)).

## 4. AES67 Stream and Session Engineering Model

### Transport and essence model

An AES67 stream in ST Lab should be modeled as an RTP audio session carrying linear PCM audio because AES67’s visible scope says high-performance audio implies linear PCM at 44.1 kHz or higher and 16-bit or higher resolution, and because AES67 lists RTP, RTP/AVP, L24 payload, and SDP dependencies as indispensable references ([AES67-2023 preview, Clauses 1-2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 3550](https://www.rfc-editor.org/rfc/rfc3550.html), [RFC 3551](https://www.rfc-editor.org/rfc/rfc3551.html), [RFC 3190](https://www.rfc-editor.org/rfc/rfc3190.html), [RFC 8866](https://datatracker.ietf.org/doc/html/rfc8866)). The base RTP header model should include payload type, sequence number, timestamp, SSRC, marker bit, and extension flag because RFC 3550 Section 5.1 defines those header fields and their semantics ([RFC 3550, Section 5.1](https://www.rfc-editor.org/rfc/rfc3550.html)).

The stream model should bind the RTP timestamp rate to the audio sample rate rather than to channel count because RFC 3551 states that the RTP clock rate for audio usually equals the number of sampling periods per second and is independent of number of channels and encoding ([RFC 3551, Section 4.1](https://www.rfc-editor.org/rfc/rfc3551.html)). The packet model should keep all channels for a given sampling instant in the same packet because RFC 3551 states that samples for all channels belonging to a single sampling instant must be within the same packet ([RFC 3551, Section 4.1](https://www.rfc-editor.org/rfc/rfc3551.html)).

L24 handling should be explicit because RFC 3190 defines L24 as an RTP payload encoding for 24-bit linear audio and defines 24-bit samples as signed two’s-complement values packed contiguously into payload octets from the most significant bit, in time sequence from the oldest sample ([RFC 3190, Section 4](https://www.rfc-editor.org/rfc/rfc3190.html)). L16 handling should be explicit but the exact AES67-2023 conformance combinations for L16/L24, packet time, and channel count are **Unverified** from primary AES text because the full AES67 Clauses 7 and Annex G PICS are not visible in the public preview ([AES67-2023 preview, Clauses 7 and Annex G table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).

### Session description model

An AES67 session description should be parsed as a structured SDP object rather than a text blob because RFC 8866 defines session-level and media-level sections, default inheritance, and ordered fields ([RFC 8866, Section 5](https://datatracker.ietf.org/doc/html/rfc8866)). ST Lab should model `m=audio`, RTP profile/protocol, RTP payload type list, RTP/RTCP ports, `c=` connection address, `a=rtpmap`, `a=fmtp`, `a=ptime`, `a=maxptime`, `a=ts-refclk`, `a=mediaclk`, and direction attributes because RFC 8866 and RFC 7273 define those fields or attributes and their relevant semantics ([RFC 8866, Sections 5-6](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 7273, Sections 4-5](https://www.rfc-editor.org/rfc/rfc7273.html)).

For multicast SDP, ST Lab should distinguish destination group address, port, TTL for IPv4, and address family because RFC 8866 states that multicast sessions use an IP multicast group address, IPv4 multicast connection addresses must include TTL, and IPv6 multicast does not use TTL scoping and must not include TTL ([RFC 8866, Section 5.7](https://datatracker.ietf.org/doc/html/rfc8866)). For unicast SDP, ST Lab should distinguish expected data source, relay, or sink because RFC 8866 states that non-multicast connection addresses contain the unicast IP address of the expected data source, data relay, or data sink as determined by additional attributes ([RFC 8866, Section 5.7](https://datatracker.ietf.org/doc/html/rfc8866)).

AES67 connection and discovery behavior must be modeled separately from RTP payload interoperability because the AES67 preview shows separate clauses for Discovery and Connection Management, and secondary guidance says AES67 does not mandate a single method for session discovery or conveying SDP ([AES67-2023 preview, Clauses 9-10 table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)). SAP can be modeled as one optional announcement mechanism because RFC 2974 defines SAP as an Experimental protocol and says all SAP implementations must support `application/sdp` payloads, but RFC 2974 does not make SAP an Internet standard ([RFC 2974, Status and Section 6](https://www.rfc-editor.org/rfc/rfc2974.html)).

## 5. Timing and Network Design Constraints

### PTP and clock-source constraints

AES67-2023 lists IEEE 1588-2008 as a normative reference and the foreword says AES67-2023 added handling for IEEE 1588-2019 in Clauses 4.0 and 8.2, so ST Lab should track both the AES67-selected PTP profile/version and the actual device-supported PTP version/profile ([AES67-2023 preview, Clause 2 and foreword](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [IEEE 1588-2008](https://standards.ieee.org/standard/1588-2008.html), [IEEE 1588-2019](https://standards.ieee.org/standard/1588-2019.html)). IEEE metadata states that IEEE 1588-2019 supersedes IEEE 1588-2008 and defines a protocol for precise synchronization of clocks in packet-based networked systems with profiles and domains, but detailed IEEE clause-level PTP requirements remain **Unverified** here because the public IEEE pages fetched do not expose full normative clauses ([IEEE 1588-2019](https://standards.ieee.org/standard/1588-2019.html)).

RFC 7273 requires ST Lab to model clock signaling as part of the SDP/session layer because it defines `ts-refclk` and `mediaclk` attributes and their inheritance levels ([RFC 7273, Sections 4.8 and 5.4](https://www.rfc-editor.org/rfc/rfc7273.html)). If PTP domains are signaled via SDP, ST Lab should check both grandmaster ClockIdentity and PTP domain because RFC 7273 says senders and receivers should check both when determining clock equivalence ([RFC 7273, Section 4.3](https://www.rfc-editor.org/rfc/rfc7273.html)).

Traceability must be a validation dimension because RFC 7273 says traceable time sources must not be mixed with non-traceable time sources at the same SDP level ([RFC 7273, Section 4.8](https://www.rfc-editor.org/rfc/rfc7273.html)). Direct-referenced media clocks must be paired with timestamp reference clock signaling because RFC 7273 says a directly derived media clock requires `a=ts-refclk` ([RFC 7273, Section 5.2](https://www.rfc-editor.org/rfc/rfc7273.html)).

### Packetization, buffering, and latency

ST Lab should compute packet time from explicit SDP `ptime` when available because RFC 8866 defines `ptime` as the length of time in milliseconds represented by the media in a packet ([RFC 8866, Section 6.4](https://datatracker.ietf.org/doc/html/rfc8866)). ST Lab should compute samples per packet as sample rate multiplied by packet time, but this formula is an engineering derivation from the RFC 8866 `ptime` definition and the RFC 3551 audio timestamp/sample-period model rather than a directly quoted AES67 formula ([RFC 8866, Section 6.4](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 3551, Section 4.1](https://www.rfc-editor.org/rfc/rfc3551.html)).

AES67-2023’s exact packet-time conformance matrix is **Unverified** from primary text because the public preview exposes Clause 7.2 headings and foreword commentary but not the actual packet-time tables or Annex G PICS criteria ([AES67-2023 preview, Clause 7.2 and Annex G table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)). Secondary AIMS PICS material for AES67-2018 states examples of 1 ms and 0.25 ms packet-time capability declarations, but this is not the AES67-2023 standard and should be treated as secondary conformance context rather than primary normative text ([AIMS AES67-2018 PICS presentation](https://www.aimsalliance.org/wp-content/uploads/2018/11/AES67-2018-PICS-a-basis-for-interoperability-assessment.pdf)).

Receiver buffering must be modeled as an implementation capability because AES67-2023 foreword says Clause 7.5 explains an interoperability exception where a lower-timing-accuracy sender connected to a receiver with only minimally required buffering cannot guarantee reliable reception of all streams ([AES67-2023 preview, foreword item for Clause 7.5](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)). The exact timing-accuracy classes, minimum buffering values, and interoperability matrix are **Unverified** until licensed AES67 Clause 7.5 and Annex G text are attached ([AES67-2023 preview, preview omissions](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).

### QoS and network treatment

QoS should be a configurable design assumption rather than a single hard-coded truth because AES67-2023 foreword states that sender default DSCP behavior was clarified in Clause 6.2, while Annex B is explicitly “Informative” and titled “Network QoS configuration recommendations” ([AES67-2023 preview, foreword and Annex B table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)). Secondary vendor guidance states that AES67 uses three traffic classes and recommends DSCP EF 46 for PTP, AF41 34 for RTP audio, and BE 0 for remaining traffic, but this should be labeled non-normative in ST Lab unless verified against licensed AES67 Clause 6.2 or Annex B ([RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).

Bandwidth modeling should separate media payload, RTP/UDP/IP overhead, RTCP overhead, and link-layer overhead because RFC 8866 says `b=AS` for RTP-based applications gives RTP session bandwidth as defined by RFC 3550, and RFC 8866 says default SDP bandwidth units include transport and network-layer overhead but not link-layer overhead ([RFC 8866, Section 5.8](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 3550, Section 6.2](https://www.rfc-editor.org/rfc/rfc3550.html)). RTCP overhead should be explicit because RFC 3550 recommends RTCP be fixed at 5% of session bandwidth unless profile/session parameters specify otherwise, while AES67-specific RTCP policy is **Unverified** from the public preview ([RFC 3550, Section 6.2](https://www.rfc-editor.org/rfc/rfc3550.html), [AES67-2023 preview, Clause 6 table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).

## 7. ST Lab Data Model Guidance (AES67-specific fields)

### Node fields

- **AES67 role**: Store sender, receiver, or both because AES67-2023 foreword says applicability of SDP output generation and input interpretation was clarified for senders and receivers, and AIMS PICS material treats audio streaming as receive/send capability dimensions ([AES67-2023 preview, foreword G.3.6.1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [AIMS AES67-2018 PICS presentation](https://www.aimsalliance.org/wp-content/uploads/2018/11/AES67-2018-PICS-a-basis-for-interoperability-assessment.pdf)).
- **PTP role and capability**: Store PTP master, slave, or both as a capability dimension because AIMS PICS material explicitly lists PTP master/slave/both as synchronization capability categories, while exact AES67-2023 PICS criteria are **Unverified** from primary text ([AIMS AES67-2018 PICS presentation](https://www.aimsalliance.org/wp-content/uploads/2018/11/AES67-2018-PICS-a-basis-for-interoperability-assessment.pdf), [AES67-2023 preview, Annex G table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **PTP version/profile support**: Store IEEE 1588-2008, IEEE 1588-2019/PTPv2.1 handling, domain, grandmaster identity, and traceability support because AES67 lists IEEE 1588-2008 and AES67-2023 foreword says IEEE 1588-2019 handling was added, while RFC 7273 uses PTP version, ClockIdentity, and domain in SDP clock signaling ([AES67-2023 preview, Clause 2 and foreword](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 7273, Section 4.3](https://www.rfc-editor.org/rfc/rfc7273.html)).
- **SDP capabilities**: Store SDP generation, SDP parsing, and SDP validation evidence because AES67-2023 foreword identifies sender output generation and receiver input interpretation requirements, and RFC 8866 defines the SDP grammar and field semantics that parsers must handle ([AES67-2023 preview, foreword G.3.6.1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 8866, Sections 5-9](https://datatracker.ietf.org/doc/html/rfc8866)).
- **Discovery and connection methods**: Store manual SDP, SAP, SIP, NMOS, vendor protocol, or other method because AES67 shows Discovery and Connection Management as separate clauses, SAP is optional/Experimental, and secondary guidance says the SDP conveyance method is not mandated by AES67 ([AES67-2023 preview, Clauses 9-10](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 2974, Status](https://www.rfc-editor.org/rfc/rfc2974.html), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).
- **QoS/DSCP capability**: Store fixed DSCP, configurable DSCP, default DSCP, management-interface override, and actual marked values because AES67-2023 foreword says default DSCP behavior was clarified as a requirement and secondary guidance says DSCP values may need network-specific treatment ([AES67-2023 preview, foreword Clause 6.2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).

### Stream fields

- **Essence tuple**: Store encoding name, bit depth, sample rate, channel count, channel order, and payload packing because AES67 scope requires linear PCM assumptions and RFC 3190/RFC 3551 define L24 packing, audio channel behavior, and RTP audio timestamp model ([AES67-2023 preview, Clause 1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RFC 3190, Sections 4 and 8](https://www.rfc-editor.org/rfc/rfc3190.html), [RFC 3551, Section 4](https://www.rfc-editor.org/rfc/rfc3551.html)).
- **RTP tuple**: Store SSRC, payload type, sequence-number space, timestamp rate, initial timestamp policy, marker-bit policy, and RTP/RTCP port pair because RFC 3550 defines RTP header semantics and RFC 8866 maps media descriptions to RTP payload types and ports ([RFC 3550, Section 5.1](https://www.rfc-editor.org/rfc/rfc3550.html), [RFC 8866, Section 5.14](https://datatracker.ietf.org/doc/html/rfc8866)).
- **Packetization tuple**: Store `ptime`, `maxptime`, samples per packet, payload bytes per packet, packets per second, and any supported packet-time set because RFC 8866 defines `ptime` and `maxptime`, while AES67-2023 packet-time conformance details are not visible in the preview ([RFC 8866, Sections 6.4-6.5](https://datatracker.ietf.org/doc/html/rfc8866), [AES67-2023 preview, Clause 7.2 table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Clock-signaling tuple**: Store `ts-refclk`, `mediaclk`, PTP version, domain, ClockIdentity, traceable flag, and inheritance level because RFC 7273 defines those attributes and their inheritance rules ([RFC 7273, Sections 4.3, 4.8, 5.4](https://www.rfc-editor.org/rfc/rfc7273.html)).
- **Session description evidence**: Store raw SDP, normalized SDP, source of SDP, trust/authentication status, and validation status because RFC 7273 warns that session descriptions cannot be trusted unless obtained from an authenticated transport protocol or trusted source ([RFC 7273, Section 7](https://www.rfc-editor.org/rfc/rfc7273.html)).

### Link and network fields

- **Addressing mode**: Store multicast or unicast because RTP supports multicast and unicast services and RFC 8866 uses different connection-address semantics for multicast and unicast ([RFC 3550, Abstract](https://www.rfc-editor.org/rfc/rfc3550.html), [RFC 8866, Section 5.7](https://datatracker.ietf.org/doc/html/rfc8866)).
- **Multicast details**: Store multicast group, source if source-specific multicast is used, IPv4 TTL where applicable, IGMP assumptions, VLAN/segment, and receiver membership because RFC 8866 requires IPv4 multicast TTL in SDP and AES67 lists IGMPv2 and IGMPv3 RFCs as normative references ([RFC 8866, Section 5.7](https://datatracker.ietf.org/doc/html/rfc8866), [AES67-2023 preview, Clause 2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Bandwidth components**: Store payload bandwidth, RTP/UDP/IP overhead, RTCP bandwidth, Ethernet/link overhead, and design headroom because RFC 8866 distinguishes SDP `b=` bandwidth semantics from link-layer overhead and RFC 3550 defines RTCP session-bandwidth behavior ([RFC 8866, Section 5.8](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 3550, Section 6.2](https://www.rfc-editor.org/rfc/rfc3550.html)).
- **QoS path**: Store DSCP values for PTP, RTP, RTCP, and best-effort traffic; switch trust/remarking policy; egress queue mapping; and whether device marking is fixed or configurable because AES67-2023 foreword identifies DSCP behavior as a clarified requirement and secondary guidance warns that device DSCP configurability varies ([AES67-2023 preview, foreword Clause 6.2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).

## 8. Validation Checklist (design-time)

- **Validate AES67 scope**: Confirm each stream is linear PCM, at least 44.1 kHz, at least 16-bit, and intended for low-latency high-performance audio because AES67 Clause 1 defines those visible scope conditions ([AES67-2023 preview, Clause 1](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Validate profile evidence**: Attach AES67-2023 licensed clause evidence, Annex G PICS, or vendor conformance statement before marking exact AES67 conformance as verified because the preview omits full Clauses 3-10 and Annex G ([AES67-2023 preview, preview omissions](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Validate RTP timing**: Check that RTP timestamp increment derives from a monotonic linear clock and that timestamp rate matches the audio sample-period model because RFC 3550 and RFC 3551 define those requirements ([RFC 3550, Section 5.1](https://www.rfc-editor.org/rfc/rfc3550.html), [RFC 3551, Section 4.1](https://www.rfc-editor.org/rfc/rfc3551.html)).
- **Validate RTP identity**: Check payload type mapping, SSRC uniqueness within the RTP session, sequence-number continuity assumptions, and receiver payload support because RFC 3550 defines payload type, SSRC, and sequence-number semantics ([RFC 3550, Section 5.1](https://www.rfc-editor.org/rfc/rfc3550.html)).
- **Validate payload packing**: Check L24 and multi-channel packing against RFC 3190/RFC 3551 rules because those sources define L24 sample packing and multi-channel same-instant containment ([RFC 3190, Section 4](https://www.rfc-editor.org/rfc/rfc3190.html), [RFC 3551, Section 4.1](https://www.rfc-editor.org/rfc/rfc3551.html)).
- **Validate SDP structure**: Parse SDP inheritance and confirm `m=audio`, `c=`, RTP payload type mapping, `a=rtpmap`, `a=fmtp`, `a=ptime`, `a=maxptime`, and direction attributes because RFC 8866 defines those structures and attributes ([RFC 8866, Sections 5-6](https://datatracker.ietf.org/doc/html/rfc8866)).
- **Validate clock signaling**: Check `ts-refclk`, `mediaclk`, PTP version, PTP domain, grandmaster ClockIdentity, traceability, and inheritance because RFC 7273 defines these fields and equivalence checks ([RFC 7273, Sections 4-6](https://www.rfc-editor.org/rfc/rfc7273.html)).
- **Validate direct media-clock dependencies**: Reject `mediaclk:direct` without a valid `ts-refclk` because RFC 7273 requires a reference clock for direct-referenced media clocks ([RFC 7273, Section 5.2](https://www.rfc-editor.org/rfc/rfc7273.html)).
- **Validate discovery method**: Confirm that sender, receiver, and controller share at least one SDP exchange or connection-management method because AES67 preview separates discovery/connection clauses and secondary guidance says AES67 does not mandate a single SDP conveyance method ([AES67-2023 preview, Clauses 9-10](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).
- **Validate QoS assumptions**: Confirm DSCP marking, switch queue mapping, and whether device marking is configurable because AES67-2023 foreword says DSCP sender defaults were clarified and secondary guidance says QoS values and configurability vary in practice ([AES67-2023 preview, foreword Clause 6.2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).
- **Validate bandwidth**: Compute payload, RTP/UDP/IP, RTCP, and link-layer overhead separately because RFC 8866’s `b=AS` and bandwidth-unit semantics do not include link-layer overhead by default ([RFC 8866, Section 5.8](https://datatracker.ietf.org/doc/html/rfc8866), [RFC 3550, Section 6.2](https://www.rfc-editor.org/rfc/rfc3550.html)).
- **Validate SAP only when selected**: If SAP is selected, check `application/sdp`, announcement scope, port 9875, and authentication assumptions because RFC 2974 defines these SAP requirements while identifying SAP as Experimental ([RFC 2974, Sections 3, 6, 8](https://www.rfc-editor.org/rfc/rfc2974.html)).

## 9. Open Questions / Unverified Items

- **Exact AES67-2023 packet-time table**: The public preview shows Clauses 7.2.1 and 7.2.2 for required and recommended packet times, but the actual tables are not visible; exact required/recommended packet times are therefore **Unverified** from primary AES67-2023 text ([AES67-2023 preview, Clause 7.2 table of contents](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Exact AES67-2023 channel-count and stream-mode criteria**: The preview shows Clause 7.3 and Annex G PICS stream-mode criteria, but full criteria are not visible; exact sender/receiver channel-count requirements are therefore **Unverified** from primary AES67-2023 text ([AES67-2023 preview, Clause 7.3 and Annex G](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Exact AES67-2023 default DSCP values**: The preview states that sender default DSCP behavior was clarified as a requirement, but it does not expose the exact default DSCP table; DSCP values from RAVENNA are secondary guidance until verified against AES67-2023 Clause 6.2 ([AES67-2023 preview, foreword Clause 6.2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf), [RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).
- **Exact AES67-2023 transport-port range**: The preview states that an exact transport-port range is defined in Clause 6.3, but the range itself is not visible; exact supported sender/receiver port range is **Unverified** ([AES67-2023 preview, foreword Clause 6.3](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Exact AES67-2023 receiver buffering and sender timing classes**: The preview states that Clause 7.5 explains a buffering/timing-accuracy interoperability exception, but the exact matrix and values are not visible; buffering requirements are **Unverified** ([AES67-2023 preview, foreword Clause 7.5](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Exact AES67-2023 handling of IEEE 1588-2019**: The preview states that IEEE 1588-2019 handling was added in Clauses 4.0 and 8.2, but detailed handling is not visible; PTPv2.1 compatibility behavior is **Unverified** ([AES67-2023 preview, foreword Clauses 4.0 and 8.2](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **Official ST 2110-30 current clause constraints**: AIMS secondary material describes ST 2110-30 as a constrained subset of AES67, but the official SMPTE page fetched did not expose ST 2110-30 clause text; ST 2110-30-specific constraints are **Unverified** for this AES67 report ([AIMS AES67/ST 2110-30 presentation](https://aimsalliance.org/wp-content/uploads/2022/04/Intro-to-AES67-and-ST-2110-30.pdf), [SMPTE ST 2110 page](https://www.smpte.org/standards/st2110)).
- **AES67-2023 printing-date conflict**: AES store metadata and the preview PDF show different printing dates, so the report stores both and does not infer a single resolved printing date ([AES Standards Store](https://aes.org/publications/standards-store/), [AES67-2023 preview](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).

## 10. Sources

- **AES Standards Store**: The AES store identifies “AES67-2023: AES Standard for Audio Applications of Networks - High-performance streaming audio-over-IP interoperability,” shows member/non-member access terms, and states that preview extracts are provided without charge as a guide to standards content ([AES Standards Store](https://aes.org/publications/standards-store/)).
- **AES67-2023 preview PDF**: The AES preview provides Clause 1 scope, Clause 2 normative references, table of contents, foreword/change notes, and document metadata for AES67-2023, but it omits full Clauses 3-10 and full Annex G PICS ([AES67-2023 preview](https://www.aes.org/tmpFiles/aessc/20250901/aes67-2023-i.pdf)).
- **IEEE 1588-2008**: IEEE metadata identifies IEEE 1588-2008 as “IEEE Standard for a Precision Clock Synchronization Protocol for Networked Measurement and Control Systems,” published 2008-07-24 and superseded by IEEE 1588-2019 ([IEEE 1588-2008](https://standards.ieee.org/standard/1588-2008.html)).
- **IEEE 1588-2019**: IEEE metadata identifies IEEE 1588-2019 as active, published 2020-06-16, and superseding IEEE 1588-2008 ([IEEE 1588-2019](https://standards.ieee.org/standard/1588-2019.html)).
- **RFC 3550**: RTP base protocol, July 2003, Standards Track, used for RTP packet, timestamp, sequence, SSRC, payload type, RTCP, jitter, and loss requirements ([RFC 3550](https://www.rfc-editor.org/rfc/rfc3550.html)).
- **RFC 3551**: RTP/AVP profile, July 2003, Standards Track, used for audio payload profile rules, audio channel ordering, audio RTP clock model, and static/dynamic payload context ([RFC 3551](https://www.rfc-editor.org/rfc/rfc3551.html)).
- **RFC 3190**: RTP payload format for 12-bit DAT audio and 20-/24-bit linear sampled audio, January 2002, Standards Track, used for L24 packing and SDP examples ([RFC 3190](https://www.rfc-editor.org/rfc/rfc3190.html)).
- **RFC 8866**: SDP, January 2021, Standards Track, used for SDP session/media structure, connection data, `rtpmap`, `fmtp`, `ptime`, `maxptime`, and bandwidth semantics ([RFC 8866](https://datatracker.ietf.org/doc/html/rfc8866)).
- **RFC 7273**: RTP Clock Source Signalling, June 2014, Standards Track, used for `ts-refclk`, `mediaclk`, PTP clock equivalence, traceability, and compatibility checks ([RFC 7273](https://www.rfc-editor.org/rfc/rfc7273.html)).
- **RFC 2974**: Session Announcement Protocol, October 2000, Experimental, used only when SAP is a selected discovery/announcement method ([RFC 2974](https://www.rfc-editor.org/rfc/rfc2974.html)).
- **AIMS AES67-2018 PICS presentation**: Secondary conformance-context source for AES67-2018 PICS concepts and example capability dimensions, not a substitute for AES67-2023 Annex G ([AIMS AES67-2018 PICS presentation](https://www.aimsalliance.org/wp-content/uploads/2018/11/AES67-2018-PICS-a-basis-for-interoperability-assessment.pdf)).
- **RAVENNA practical AES67 guide**: Vendor/secondary source used only for operational risks around discovery, DSCP practice, and implementation divergence, not for primary normative requirements ([RAVENNA practical AES67 guide](https://www.ravenna-network.com/your-practical-guide-to-aes67-part-1-2/)).
- **AIMS AES67/ST 2110-30 presentation**: Secondary ecosystem source used only to flag that ST 2110-30 may constrain AES67 behavior in broadcast contexts, not to establish SMPTE normative requirements ([AIMS AES67/ST 2110-30 presentation](https://aimsalliance.org/wp-content/uploads/2022/04/Intro-to-AES67-and-ST-2110-30.pdf)).
- **SMPTE ST 2110 page**: Official SMPTE educational/resource page fetched for ST 2110 context; no ST 2110-30 clause-level normative text was visible in the fetched page ([SMPTE ST 2110 page](https://www.smpte.org/standards/st2110)).
